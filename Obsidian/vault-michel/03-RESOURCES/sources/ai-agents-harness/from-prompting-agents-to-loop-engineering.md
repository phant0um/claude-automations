## Tese central
Uma afirmação circulando em círculos de AI coding: parar de prompt-ar agentes de código e começar a desenhar loops que os promptam por você. Peter Steinberger: "You shouldn't be prompting coding agents anymore. You should be designing loops that prompt your agents." Boris Cherny (criador do Claude Code) confirma do outro lado: "I don't prompt Claude anymore. I have loops that are running... My job is to write loops." O artigo argumenta que prompt engineering não está morto — o trabalho sobe um nível, de escrever o código para escrever o sistema que escreve o código.

## Argumentos principais
- Um loop é um pequeno programa que faz quatro coisas: prompta o agente de código, lê o que ele produziu, decide se terminou, e se não, prompta de novo com o erro ou o próximo passo. A forma é sempre a mesma: definir um objetivo, agir, checar, alimentar o erro de volta, repetir até o check passar ou o loop se parar.
- "Loop" significa pelo menos cinco coisas diferentes, e boa parte da discordância vem de pessoas usando uma palavra para cinco ideias: ReAct (2022, padrão de pesquisa original: reason-act-observe-repeat) → AutoGPT (2023, loop de objetivo auto-promptado, notório por não saber quando parar) → ralph loop (reset deliberado de contexto entre iterações para o agente não se afogar na própria história) → `/loop` e `/goal` (cadência e condições de conclusão embutidas no agente, carregando estado entre turnos) → orquestração (um autor abre leque para muitos agentes que leem GitHub/Slack/chat e decidem o que construir a seguir).
- A mudança de onde o custo se acumula: por dois anos a pergunta de custo era "qual modelo, quantos tokens" — dentro de um loop, isso aponta para a camada errada. O gasto não é mais uma chamada única, é quantas vezes o loop dá a volta — "um loop que tenta de novo seis vezes antes de convergir custa seis vezes mais que um que acerta de primeira, no mesmo modelo."

## Key insights
- **As seis partes que toda loop monta**: um trigger (algo que inicia sem você apertar go: schedule, webhook, mudança de arquivo, label numa PR — isso é o que separa um loop real de uma execução única repetida à mão); isolamento (checkout privado por agente, geralmente git worktree, para que dois agentes simultâneos não sobrescrevam os arquivos um do outro — para de ser opcional ao rodar mais de um); contexto escrito (convenções, passos de build, regras específicas do projeto mantidas onde o agente lê a cada execução — sem isso o loop re-deriva o projeto do zero a cada passada); alcance às ferramentas (conectores ao issue tracker, CI, banco, chat, para o loop abrir a PR, linkar o ticket, postar o resultado em vez de imprimir um fix e esperar você carregar o resto); um segundo agente que checa (um worker separado que avalia o output, mantido apartado de quem o produziu, porque um modelo revisando seu próprio trabalho passa quase tudo); estado em disco (markdown, board, ou queue — qualquer coisa fora da conversa que registra o que está terminado e o que vem a seguir, porque o modelo esquece entre execuções e o arquivo não).
- **Exemplo concreto "PR babysitter"**: trigger a cada 15 minutos; scope = PRs abertas com label `agent-watch`; ação = se CI está vermelho por razão determinística, tenta um fix; se main moveu, faz rebase uma vez; budget = uma tentativa de fix por PR, cinco minutos, dez arquivos alterados; stop condition = CI verde, ou budget esgotado, então para e avisa um humano. A mesma forma cobre health de CI (agrupar runs falhos por assinatura a cada 30min), verificação de deploy (bater nos endpoints depois de um push, confirmar 200s e conteúdo esperado), e clustering de feedback (agrupar comentários de canais em temas a cada 30min).
- **`/goal` em Claude Code**: você dá um end-state verificável e ele continua dando turnos até esse estado ser verdade — `/goal tests in test/auth pass` dentro de uma sessão lançada. Um `/goal` forte lê menos como um prompt e mais como um contrato — especifica quatro coisas: o **end state** desejado, a **evidência** que prova que foi alcançado, as **constraints** que o agente não pode violar no caminho, e o **budget** de trabalho permitido. Deixar qualquer um vago e o modelo preenche a lacuna com a leitura mais fácil: para antes, atalha, ou redefine sucesso para a transcrição parecer terminada enquanto o sistema real está quebrado.
- **As 4 etapas de `/goal`**: definir a condição (`/goal` + end-state verificável); o agente trabalha um turno (edita, roda testes, mostra resultados na sessão); um avaliador checa (um modelo rápido lê a transcrição e decide se foi atingido — o agente não avalia o próprio trabalho); loop ou termina (não atingido = outro turno com orientação; atingido = o goal se encerra e a execução para). A camada de avaliador esconde uma sutileza útil: o checker não precisa ser o mesmo modelo do coder — uma vez que o loop tem papéis distintos (planner, executor, evaluator, vision reviewer), cada um pode rodar num modelo diferente, e escolher qual modelo enche qual papel se torna decisão de arquitetura, não uma única aposta no "melhor" agente de código.
- **Controles que mantêm `/goal` confiável**: checar de forma mensurável (resultado de teste, exit code, contagem de arquivo, queue vazia — "npm test exits 0" é um goal, "make it better" não é); limitar a execução ("ou pare depois de 20 turnos" para que um loop travado pare em vez de queimar turnos); pareie com auto mode para turnos rodarem sem supervisão, e use `/goal clear` para abandonar antecipadamente.
- **Cinco passos do setup de Boris Cherny para rodar Opus autonomamente por horas**: (1) auto-aprovar permissões para o agente não parar para perguntar em toda chamada de ferramenta; (2) usar dynamic workflows (Ultracode no prompt) para abrir leque entre muitos agentes em vez de uma thread serial; (3) usar `/goal` ou `/loop` para manter rodando — `/goal` define condição de conclusão, `/loop` reprompta numa cadência, ambos carregam estado; (4) rodar na nuvem (desktop ou app mobile) para a sessão sobreviver ao fechar o laptop; (5) dar uma forma de se auto-verificar end-to-end — Claude in Chrome para web, um simulator MCP para mobile, e um servidor live para backend — "this is the step that makes the other four safe."
- **crabfleet (Peter Steinberger, projeto OpenClaw)**: "mission control for agent runs" — loop empacotado como produto. Trabalho como cards num board (entrados a partir de prompt, GitHub issue, ou PR, movendo por todo/running/human review/done — o board é a queue do loop e o passo de stop-and-report, tornado visível); execuções duráveis, não fire-and-forget (cada execução é uma tentativa rastreada com heartbeats, sobrevivendo a um laptop fechado); agentes que geram agentes (uma execução pode iniciar sessões filhas, enviar mensagens, ler transcrições, e atualizar seu próprio resumo de dentro de um sandbox).
- **Quando NÃO fazer loop**: edições one-shot (se você termina numa única passada, um loop é puro overhead); trabalho não-escopado ou exploratório ("descubra por que usuários estão saindo" não tem condição de aprovação, então o loop nunca converge); qualquer coisa sem um check automatizado barato (se o único verificador são seus próprios olhos, você ainda está dentro do loop — construa o check primeiro, ou faça a tarefa à mão).
- **O que pode dar errado**: o fardo de verificação permanece humano (o loop escreve mais rápido do que você consegue revisar — se você parar de ler os diffs, não removeu o trabalho, só o adiou); gaps de compreensão se ampliam (shippar código que você não escreveu, mais rápido do que consegue absorver, erode o modelo mental do seu próprio sistema, e essa dívida vence no próximo incidente); drift silencioso num check fraco (um verificador frouxo deixa passar trabalho errado-mas-aprovado em toda iteração, então o loop parece produtivo enquanto cava um buraco).
- **Como construir o seu próprio**: (1) escolher uma tarefa repetível; (2) escopar apertado ("fix the billing webhook validation, only touch app/api/billing and lib/billing" vence "fix the bug" — um loop solto vagueia); (3) dar budget e stop condition (max attempts, max runtime, max files, max spend, max consecutive failures); (4) adicionar um verificador independente (um subagente separado avalia o trabalho, porque o agente que escreveu o código é o pior juiz de se está terminado); (5) rodar numa cadência (`/loop` para intervalo, cron para schedule, hooks em pontos do ciclo de vida, ou GitHub Actions); (6) manter memória em disco (markdown ou board, não na janela de contexto).

## Exemplos e evidências
- Citações primárias: Peter Steinberger (@steipete, 7 jun 2026, 2.2M views) e Boris Cherny (@bcherny), ambos linkados no artigo original.
- Sequência de comando completa citada para rodar Opus autonomamente: `claude --permission-mode auto` → `ultracode orchestrate sub-agents...` → `/goal all tests pass and the demo loads clean` → cloud/desktop app → chrome ext/sim MCP/live server.
- crabfleet citado como exemplo real de "loop hardened into infrastructure: a queue, durable execution, fan-out, and a human-review gate are now things you configure rather than hand-script every time."
