## Central Thesis

When an AI agent deletes a production database, the model is almost never the weak link. The dangerous part is rarely the code the agent writes — it's what sits around that code, the access it inherits, and the boundary nobody put in front of it. A coding agent reasons about correctness, not operational safety. The solution is to move controls from the app to the runtime platform, where credentials are scoped server-side, access runs through shared permission groups, and writes are gated and logged centrally.

## Key Arguments

### A Coding Agent Reasons About Correctness, Not Operational Safety

A coding agent optimizes for whether the code works. It has no signal for the runtime context: who is calling it, whether they're allowed to, what the blast radius is when a call is wrong. Those constraints live outside the prompt and the codebase, so they never reach the output.

How bad a wrong call gets depends on what it does:
- An unauthorized **SELECT** exposes data that should have stayed private
- An unauthorized **write** (UPDATE, INSERT, DELETE) changes the state of the business — once committed, there is no undo

A better model won't solve this. The agent only has the code in front of it. Identity, permissions, and logging live outside the code, in the system that runs it.

### Test Run

Asked Claude Code to build a small internal support console with one button that issues an account credit. The prompt was detailed (layout, UI quality, data layer) but said nothing about authorization. It worked on the first run — the credit was applied instantly on click. The agent generated the code end-to-end and stopped. Any caller could reach it, no role decided whether they were allowed, and nothing was recorded.

The model knows a button moving money in production needs an access check — it just didn't add one because the prompt didn't ask it to.

### Why Does This Happen?

In one widely-reported case, eleven ALL-CAPS warnings not to touch production did not stop an agent from deleting the database. Instructions live inside the same context the model is free to override — they are **guidance, not enforcement.**

Ideally, controls should live in the platform the app runs on, not in the app. The moment each app handles its own permissions, you get a hundred tools with a hundred different rules and no central way to manage them. That sprawl is what **Shadow AI** really means.

### Solution: Runtime-Level Enforcement

The boundary should belong to the runtime where:
- Credentials are scoped server-side
- Access runs through shared permission groups
- Writes are gated and logged centrally

**Demonstration with Retool:** Took the same React app (generated by Claude Code with no knowledge of Retool) and dropped its bundle into Retool. Retool parsed the components, pulled them in, and walked through pointing the app's data calls at a Retool resource — the governed connection that fronts the real database.

Once data calls ran through the resource layer:
- The app inherited identity, permissions, and an audit trail from the runtime
- **SSO** resolved the caller to a real identity
- **Permission groups** decided whether the role can issue credit
- The mutating query trips an **approval gate** (on by default) — the write waits for a human instead of committing silently
- Each run lands in an **audit log** with a name and timestamp

Nothing in the app changed. The same credit button with the same function behind it. What changed is the write now runs through Retool's boundary.

The controls don't live in the code the agent wrote but directly in the environment it runs in, so they hold even when whoever built the app never knew to add them.

## Key Insights

- **Instructions are guidance, not enforcement.** ALL-CAPS warnings in prompts are overridable by the model. Real enforcement must live in the runtime, not the prompt.
- **A coding agent optimizes for correctness, not safety.** It generates the operation and leaves out everything that governs it — identity, permissions, logging, approval gates.
- **Writes are categorically different from reads.** An unauthorized SELECT exposes data; an unauthorized write changes the state of the business permanently.
- **Shadow AI = permission sprawl.** When each app handles its own permissions, you get a hundred tools with a hundred rule sets and no central management.
- **The boundary belongs to the platform, not the app.** Build wherever you're fastest (Claude Code, Cursor, Codex) and let the runtime supply the boundary on the way to production.
- **The app was Retool-unaware and still imported cleanly.** Nothing in the build was written for Retool, yet the runtime enforced identity, permissions, data access, approvals, and logging automatically.
