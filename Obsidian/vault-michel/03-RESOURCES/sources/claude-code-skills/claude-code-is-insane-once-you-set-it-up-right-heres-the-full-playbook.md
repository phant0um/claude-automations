## Tese central
Um breakdown completo do "playbook" que Boris Cherny (criador do Claude Code) demonstrou em um vídeo de 28 minutos publicado em 22 de maio de 2025 (1.3 milhões de views, zero legendas, redescoberto pelo AI Twitter a cada poucos meses). A ideia central é o que Boris chama de **compound engineering**: cada erro vira uma regra permanente, cada tarefa repetida vira um comando reutilizável, cada tarefa longa vira um sub-agente, cada passo de verificação vira um hook. O sistema fica mais inteligente a cada sessão porque o contexto, os comandos e as rotinas vivem dentro do próprio codebase — e o setup dele é "surpreendentemente vanilla": a alavancagem vem de cinco coisas empilhadas sobre o Claude Code padrão.

## Argumentos principais
- **Os cinco primitivos do playbook** (a "operating system" completa para trabalhar com Claude Code):
  1. Contexto persistente através de arquivos `CLAUDE.md`
  2. Workflows reutilizáveis através de slash commands e sub-agentes
  3. Pós-ações determinísticas através de hooks
  4. Execução paralela através de git worktrees
  5. Disciplina de "plan-first, verify-always"
- **"This is suddenly everywhere again because Hermes Agent swarms, SOUL.md templates, and Claude Routines all run on these exact 5 primitives. The orchestrators changed. The foundation didn't."** — a tese de que a fundação se mantém estável enquanto os "wrappers" (orquestradores) mudam a cada ciclo de hype.
- **Step 1 — CLAUDE.md como memória institucional viva**: Boris mantém três versões — um `CLAUDE.md` de raiz do projeto, versionado em Git e compartilhado pelo time, atualizado várias vezes por semana; um `~/.claude/CLAUDE.md` de nível de usuário para preferências pessoais entre projetos; e um `CLAUDE.local.md` que fica fora do Git para overrides pessoais. A linha-chave de Boris: "Anytime we see Claude do something incorrectly we add it to the CLAUDE.md, so Claude knows not to do it next time" — todo erro é um imposto de uma vez só; adicione a regra, nunca pague de novo.
- **Tamanho importa — "trim ruthlessly"**: o CLAUDE.md pessoal de Boris tem ~100 linhas / ~2500 tokens. Arquivos inflados são um anti-padrão real — quanto maior, mais diluem as regras que de fato importam. `/init` dentro do Claude Code escaneia o codebase e gera um arquivo inicial, que se edita ao vivo enquanto se trabalha.
- **Step 2 — slash commands para trabalho de inner-loop**: receitas de workflow reutilizáveis armazenadas como Markdown em `.claude/commands/`, invocadas com `/commandname`. Tudo que se faz cinco vezes por dia merece entrar ali. Não se escreve do zero — pede-se ao Claude para escrever o arquivo de comando, descrevendo o workflow uma vez. Exemplos de Boris: `/commit-push-pr` (estagia mudanças, escreve commit, faz push, abre PR — usado diariamente), `/techdebt` (roda no fim de uma sessão, encontra duplicação, código morto, inconsistências), comandos de "context-sync" (puxam o estado mais recente de arquivos relacionados antes de uma mudança multi-arquivo).
- **Step 3 — sub-agentes para separação de responsabilidades**: um sub-agente é uma instância especializada do Claude com seu próprio papel, prompt e instruções, vivendo como arquivo Markdown em `.claude/agents/`. A sessão principal os chama "como funções" — o sub-agente roda isolado, termina seu trabalho, devolve o resultado, e o contexto principal permanece limpo. Exemplos demonstrados por Boris: `code-simplifier.md` (roda no diff final e remove qualquer "esperteza" que o Claude over-engineerou), `verify-app.md` (testes end-to-end completos na feature recém-lançada), `build-validator.md` (confirma que o build passa em todo ambiente suportado), `code-architect.md` (usado em modo de planejamento para pensar decisões estruturais antes de qualquer código ser escrito). Podem ser chamados explicitamente em paralelo ("use 3 parallel sub-agents to brainstorm three different approaches... compare them. Pick the best.") ou encadeados (plan → coder → verifier → simplifier).
- **Step 4 — hooks para pós-ações determinísticas**: um hook é uma ação determinística que dispara em um evento do Claude Code, configurado nas settings do projeto. Os dois eventos mais usados por Boris: `PostToolUse` (dispara após o Claude editar um arquivo — Boris roda `bun run format` aqui para pegar problemas de formatação antes que cheguem ao CI) e `Stop`/`agent-stop` (dispara quando um agente termina sua tarefa — Boris roda verificação: testes, type checks, o que precisa passar antes de confiar no output). A diferença entre hook e slash command é "quem está no comando": comandos você invoca, hooks disparam automaticamente. Configura-se uma vez, roda para sempre.
- **Step 5 — sessões paralelas com git worktrees**: o maior unlock de produtividade do vídeo, segundo o próprio Boris ("the thing his team gained the most from"). Setup: três a cinco abas de terminal, cada uma um git worktree diferente apontando pro mesmo repo mas branches diferentes, cada uma rodando sua própria sessão do Claude Code, cada sessão trabalhando em uma tarefa diferente ao mesmo tempo (uma feature nova, testes da feature anterior, refatoração de módulo antigo, documentação, debug de produção). Boris também mistura claude.ai/code na web e no app mobile — começa no desktop, passa para web, checa progresso pelo mobile; o agente não se importa com qual cliente você está usando. Worktrees importam por **isolamento de conflitos**: cinco sessões escrevendo no mesmo checkout colidem; cinco sessões em cinco worktrees não.
- **Step 6 — plan first, verify always, then execute**: a última peça é uma disciplina, não um arquivo. Citações diretas de Boris: "A good plan is really important to avoid issues down the line" e "Give Claude a way to verify its work. If Claude has that feedback loop, it will 2-3x the quality of the final result." Workflow recomendado: começar toda tarefa não-trivial em modo de planejamento (Shift+Tab duas vezes — modo somente-leitura, sem edições, itera-se no plano até ficar sólido, depois muda para auto-accept e o Claude "one-shots" a implementação); sempre dar ao Claude uma forma de checar o próprio trabalho (testes, screenshots de browser para UI, simuladores para mobile, type checker, linter — qualquer feedback loop é melhor que nenhum); quando algo dá errado, replanejar em vez de insistir no plano quebrado. "The prompt is the last step, not the first. 80% of the work happens before you tell Claude to execute."
- **Quando esse playbook funciona / não funciona**:
  - Use quando: você está enviando código de produção com Claude Code mais de uma vez por semana; seu time tem mais de um engenheiro usando Claude; você corrige os mesmos erros do Claude repetidamente; seu codebase tem convenções reais que valem a pena impor; você está enviando múltiplas features em paralelo.
  - Pule quando: você está fazendo "vibing" em um projeto de fim de semana sem se importar com consistência; usa Claude para scripts avulsos que nunca tocam um repo real; o codebase é tão pequeno que um CLAUDE.md parece mais pesado que o próprio código.
- **Quatro avisos honestos antes de rodar**: bloat do CLAUDE.md é real (um arquivo de 500 linhas dilui o que importa — corte sem piedade, cada regra deve ter "ganho seu lugar"); sub-agentes não são de graça (cada chamada queima tokens — não encadeie seis sub-agentes em uma tarefa que uma sessão principal bem-prompted resolveria); hooks podem mascarar problemas (um hook PostToolUse que auto-formata pode esconder bugs sutis na forma como o Claude está estruturando código — leia seus diffs); sessões paralelas exigem cabeça clara (cinco sessões em cinco worktrees só é poderoso se você consegue rastrear o que cada uma está fazendo — comece com duas, adicione mais conforme aprende a orquestrar).

## Key insights
- **A frase mais citável de todo o artigo, e talvez do batch inteiro**: "The model isn't the problem. Opus is the best coding model that's ever shipped. The problem is the workflow." — recoloca o foco do debate de "qual modelo é melhor" para "qual sistema você construiu ao redor dele", ecoando exatamente a tese central de [[03-RESOURCES/sources/autoresearch-your-way-into-improving-your-models-harness-how-we-beat-r]] (harness × model).
- **Compound engineering é, na prática, o mesmo loop de auto-aperfeiçoamento da Camada 4 descrito em [[03-RESOURCES/sources/anthropic-says-claude-writes-80-of-its-code-here-are-4-layers-and-7-st]]** — só que feito manualmente, sem journal automatizado: cada erro vira uma regra (= "promote" após "trace" mental), cada tarefa repetida vira comando, cada verificação vira hook. O playbook de Boris é, essencialmente, o "loop de 15 minutos" do artigo da Anthropic praticado informalmente desde 2025.
- **O exemplo concreto de empresa real (@ignytlabs)** dá números operacionais: CLAUDE.md de ~90 linhas ("every line is a scar from a real mistake on a real client project"), ~12 slash commands no MVP starter padrão (com `/plan`, `/ship`, `/audit`, `/sync` usados todo dia), e média de 3 sessões paralelas por projeto de cliente ativo — "same quality, same scope, fewer calendar days. The bottleneck moved from prompting to planning."
- **A distinção operacional entre slash command e hook**: "who's in charge" — comandos são invocados (você decide quando), hooks disparam automaticamente (o sistema decide). Essa distinção é uma lente útil para classificar qualquer mecanismo de automação em harness design.
- **A virada de "panic prompting" para "architecting"**: o vídeo descreve o problema de "vibe coding" com precisão clínica — sessão começa do zero contexto, output deriva, pela manhã o Claude esqueceu tudo que você ensinou ontem. A solução de Boris não é "prompt melhor", é "sistema melhor" — o prompt vira o último passo, não o primeiro.
- **"This isn't a prompt list... it's a full operating system for working with Claude Code"** — posiciona o playbook como infraestrutura, não dica avulsa, alinhado com a metáfora de "AI operating system" já presente no vault ([[03-RESOURCES/concepts/agent-systems/ai-operating-system]]).

## Implicações para o vault
- Esta fonte é a **referência primária canônica** para o conjunto de práticas que [[03-RESOURCES/concepts/agent-systems/agentsmd-pattern]], [[03-RESOURCES/concepts/agent-systems/handoff-file-pattern]] e [[03-RESOURCES/concepts/agent-systems/claude-code-skills]] já documentam de forma fragmentada — vale enriquecer essas páginas com a estrutura dos "5 primitivos" e a citação de origem (Boris Cherny, vídeo de maio/2025).
- Conecta-se diretamente com [[03-RESOURCES/concepts/agent-systems/subagent-pattern-empirical]] e [[03-RESOURCES/concepts/agent-systems/subagent-spawning]]: os exemplos de sub-agentes especializados (`code-simplifier`, `verify-app`, `build-validator`, `code-architect`) são instâncias concretas e nomeadas que podem servir de catálogo de referência.
- A disciplina "plan-first, verify-always" é uma confirmação prática e anterior (maio/2025) do mesmo princípio que aparece formalizado oficialmente em [[03-RESOURCES/sources/build-an-orchestration-mode]] (modo de planejamento read-only = Shift+Tab; verificação adversarial = sub-agente verificador). Ambas as fontes convergem na ideia de "scout/plan inline → fan out → verify".
- O conceito de **compound engineering** ("every mistake becomes a permanent rule, every repeated task becomes a reusable command, every long task becomes a sub-agent, every verification step becomes a hook") é estruturalmente idêntico ao loop trace→propose→validate→promote do artigo da Anthropic — vale anotar essa equivalência em [[03-RESOURCES/concepts/agent-systems/agent-feedback-loop-learning]] ou [[03-RESOURCES/concepts/agent-systems/self-evolving-agents]] como "versão manual, pré-tooling, do mesmo loop".
- A observação "the orchestrators changed, the foundation didn't" — citando Hermes Agent swarms, SOUL.md templates, e Claude Routines como reimplementações dos mesmos 5 primitivos — é relevante para [[03-RESOURCES/concepts/agent-systems/hermes-agent-architecture]] e merece nota cruzada lá.
- O alerta "CLAUDE.md bloat is real... trim ruthlessly" reforça diretamente uma prática já adotada no próprio vault-michel (CLAUDE.md do projeto está estruturado em seções enxutas com invariantes marcados) — confirmação externa de uma decisão de design já tomada, não contradição.
- Não foram identificadas contradições com páginas existentes — esta fonte consolida e dá proveniência (com citações diretas e datas) a práticas que o vault já documenta de forma mais dispersa.
