## Tese central
Prompting has become the primary interface between humans and generative AI, yet many natural language prompts remain fragile: roles, goals, constraints, and expected outputs are often buried in prose or left implicit. In agentic and software development workflows, a misread at the first handoff can propagate through every step, since a significant portion of agent failures stem from context ambiguities rather than model limitations. ## Argumentos principais
### 1 Introduction
The telephone game is a familiar version of what psychology calls serial reproduction, where a message changes as it passes through a chain of interpretation [^5]. In that setting, the distortion occurs between humans and grows with every retelling. In the AI era, generative AI has made human-to-AI interaction a much more common part of Human-Computer Interaction (HCI), with people asking Large Language Models (LLMs) and agentic systems to infer goals, constraints, priorities, and acceptable outputs from ordinary language. The first handoff is where this distortion enters: if intent is not conveyed clearly at the start of the interaction, the model commits to a flawed reading, and agentic systems can carry it forward and amplify it at each step, much as later players repeat a message that has already drifted. In daily work and especially across the software development lifecycle, the quality of that initial inference often decides whether AI support saves time or creates rework.
LLMs are useful because they can interpret context, summarize information, generate structured content, write code, plan steps, and call tools. Recent models span billions to trillions of parameters, from 8B and 70B systems to larger 405B and near-trillion-parameter models. Parameter count does not directly equal stored knowledge, but it is a proxy for representational capacity: Morris et al. [^18] estimate that GPT-family models hold an approximate capacity of 3.6 bits per parameter. By that measure, a 70B-parameter model may encode on the order of tens of gigabytes (roughly 30 GB) of compressed, distributed knowledge. This knowledge is not exposed as a clear database that a user can inspect; it lives in model weights as statistical patterns and is retrieved probabilistically. The prompt, therefore, becomes the main interface for selecting and shaping what the model should do. Effective prompting is more than polite wording. It is the practical act of translating human intent into a form that a model can recognize, organize, and execute [^7] [^14].
Current prompt practice still relies heavily on free-form prose. Guides commonly recommend roles, background context, examples, constraints, output formats, and iteration, and context engineering frames these choices as the work of supplying the right information at the right time [^10] [^19]. Yet in real tasks these elements are often buried inside long paragraphs, repeated in slightly different forms, or left implicit. This is costly for beginners because they may not know which pieces matter, and it is also awkward for experienced software practitioners because their working mindset is usually shaped by explicit interfaces, requirements, types, and control structures rather than open-ended semantic guessing.

### 2 Related Works
Prompt engineering is now an established practice with a documented toolkit of techniques, such as role prompting, chain-of-thought, few-shot exemplars, and explicit constraints [^7] [^14] [^6] [^15]. Liu et al. [^14] sort these into four aspects, namely profile and instruction, knowledge, reasoning and planning, and reliability, and note that unstructured prompts tend to give variable, less reliable answers. Li et al. [^13] tie the fragility of manual prompting to expert dependency and format sensitivity, where small shifts in punctuation or phrasing change the result. Beginner guides agree, since a good prompt has to set a role, limit the scope, and fix the output format [^6] [^7], yet they leave this as advice rather than enforceable structure. PromptMN recasts these levers as typed keywords like %role, %mustnot, and %format, so the recommended structure becomes explicit and checkable.
Context engineering moves the focus from the model to the input it receives. Savva [^19] describes context provision, that is problem framing, information selection, constraints, and evaluation criteria, as a costly input that hinges on expertise and is hard to observe, and argues that this context, not raw model capability, sets the value of the output. Hua et al. [^10] define the practice as designing and organising context so machines act in line with human intent, and trace it over a twenty-year path from human-computer interaction to today’s agents. The evidence backs this up, because Adam et al. [^1] report that screening F1 swung widely with prompt composition and improved when prompts named the task and listed selection criteria. Gibreel et al. [^9] read prompt-engineering competence as the skill of stating goals, constraints, and context clearly, while MEMENTO [^11] shows that a model’s own reasoning benefits from being split into blocks and compressed. These findings motivate PromptMN’s intent and requirement clusters, which reserve a fixed slot for framing, constraints, and success criteria.
The need for structure grows once prompts begin to drive autonomous, multi-step work. Practical guides cast an agent as a system that reasons, plans, and acts toward a goal with limited human involvement [^2], and production reports favour workflow decomposition, deterministic orchestration, and externalised prompts over one monolithic prompt [^4]. Pattern catalogues and SDLC reviews repeat the same parts, such as prompt chaining, evaluator and reflect-refine loops, and role specialisation with iterative feedback [^3] [^20]. ReAct [^22] adds that weaving reasoning traces through actions curbs hallucination and error propagation while keeping each step auditable. PromptMN moves these primitives into the prompt through ordered steps and plan disclosure (%plan, %showplan, %trace), control flow (%if, %repeat, %method), and explicit data boundaries (%in, %out).

### 3 Methodology
This section presents the design and structure of PromptMN. It opens with the rationale for giving prompts a more defined form, then describes the syntax and keyword set that make up the language, explains how those keywords are grouped into functional clusters, and closes with the contract that governs how a model is expected to process PromptMN directives.

### 3.1 Design Rationale
Prompt engineering has been developing from informal experimentation into a recognised discipline with documented techniques, including role prompting, chain-of-thought reasoning, few-shot demonstrations, and explicit constraints [^7] [^14].Yet the practice has remained largely unstructured: prompts take the form of free-form natural language instructions, with a notable lack of common vocabulary, consistent formatting conventions, or any mechanism for systematic adaptation across tasks and contexts [^13]. As prompts grow to coordinate multi-agent workflows, their implicit structure becomes a bottleneck, which motivates the case for integrating explicit prompting frameworks rather than relying on unstructured free-text instructions [^20].
Context engineering research has made the cost of this informality visible. As observed by Philipp Schmid [^19], a substantial subset of agent failures can be classified as context failures rather than fundamental model failures, underscoring the need for structured prompt markup. Hua et al. [^10] trace context engineering from its roots in sensor-driven, rule-based HCI systems, where designers acted as "intention translators", through to today’s agentic pipelines, where LLMs are still fundamentally bottlenecked by the quality of input tokens rather than model capacity alone. Work on building production agentic systems [^4] traces brittle pipelines and non-deterministic failures back to weak task decomposition and agents carrying too many responsibilities, while ReAct [^22] showed that weaving reasoning traces alongside actions drops hallucination rates and keeps decision steps auditable. Yet none of these efforts appears to provide a formal, reusable vehicle for expressing that structure.
PromptMN addresses this gap in part. It introduces a pseudo-prompting domain-specific language (DSL) whose reserved keywords impose sufficient structure for prompts to be parsed, reviewed, and reused, while leaving all prose content deliberately unconstrained. The goal is not to replace natural language but to annotate it with typed, mnemonic directives.

### 3.2 Language Design
PromptMN keywords are written inline with natural language and prefixed with ‘%‘. They act as typed directives that an LLM or agentic runtime is expected to recognise and resolve. Statements are terminated by semicolons (;), and related statements can be grouped into block scopes with curly braces ({…}). Newlines (\\n) mark the start of a new term or expression.
A central design decision is semantic resolution over source order: the interpreter reorders directives by role before execution, so authors may compose prompts in any reading order without affecting interpretation. By relying on the natural language understanding capabilities of the underlying model, this approach relaxes the authoring burden while supporting a more predictable and well-defined semantic interpretation.
The vocabulary is intentionally small, comprising forty-eight named tokens plus the numeric step pattern %\<number>, and every token is mnemonic (%plan, %req, %if, %out). Keeping the surface area small lowers the entry cost for writers, echoing prior guidance that prompts work best when they are concise and unambiguous, while wordy or vague phrasing tends to produce weaker outputs [^6] [^7]. The delimiter pair $\infty$ … $\infty$ marks the boundary of a PromptMN block, making it easy to embed programs inside larger documents or system prompts. Such explicit declarations also help an interpreter reliably recognise PromptMN keywords within surrounding natural-language context, avoiding accidental capture of similar words in the user’s prose.

### 3.4 Interpreter Contract
PromptMN does not prescribe a specific model or execution engine. The interpreter contract requires only that its specification be present within a context window or system prompt, or that an LLM be fine-tuned on PromptMN for its proper recognition. This loose coupling means any sufficiently capable LLM can serve as the runtime, and compliance can be tested with a small suite of functional prompts whose observable outputs, namely the externalized execution plan from %showplan and the post-hoc reasoning log from %trace, provide concrete, inspectable evidence of correct directive resolution.
The language is intentionally forward-compatible: %newconcept lets practitioners introduce domain-specific keywords without modifying the base specification, and %meta provides a hook for runtime-specific extensions. Both mechanisms reflect the observation that context engineering practices must evolve alongside model capabilities. Both mechanisms reflect the observation that Hua et al. [^10] place at the center of their framework: the techniques and formats of context engineering must change as machine intelligence advances.

### 4.2 Reverse Prompt Engineering with PromptMN
Reverse prompt engineering (RPE), also known as reverse prompting, flips the conventional workflow on its head: instead of crafting a prompt and hoping the model interprets it correctly, you supply the desired output and ask the model to work out what prompt would most reliably reproduce it. Rather than relying on a single, static instruction written upfront, this retrospective loop lets the model continuously refine its contextual understanding by reasoning backward from results. Femepid et al. demonstrate that this self-correcting dynamic leads to measurable, compounding improvements in output accuracy over time [^8].
RPE pairs particularly well with PromptMN. Because PromptMN prompts are structured, typed, and semantically ordered, with each keyword carrying an unambiguous intent, the model has a far cleaner signal to reason from than it would with free-form prose. Tracing which directive produced which part of the output becomes straightforward, making the backward-reasoning loop faster, more precise, and more actionable.

### 5 Conclusion
PromptMN proposes a compact domain-specific language for a problem that now sits at the centre of human-to-AI interaction: how to express intent clearly enough that a generative model can act on it, preserve it, and expose how it was interpreted. Prompting is no longer only a matter of writing persuasive natural-language instructions. In agentic and software-development settings, a prompt increasingly behaves like an executable handoff. It carries role, goal, context, constraints, priorities, data boundaries, planning steps, and expected outputs. When these elements remain buried in free-form prose, an early misunderstanding can move through the workflow before a human can see where the drift began.
The contribution of PromptMN is to make that handoff explicit without asking users to become programmers. Its %-prefixed directives provide a small shared vocabulary for structuring ordinary language: %role establishes stance, %goal defines the target, %req, %should, %could, and %mustnot make priorities visible, %plan and %showplan surface the intended sequence of work, and %in and %out clarify data boundaries and expected results. The language does not replace prose. It gives prose a typed frame, so both the author and the model can inspect the same instruction with less ambiguity.

### Impact on Human-to-AI Interaction
Typed, semantically ordered directives give a model a cleaner signal than loose prose. They make it easier to see what the user meant and which part of the instruction shaped which part of the response. %showplan externalizes the assembled plan before work begins, while %trace can return a reasoning log afterward. Together, they create review points at both ends of the exchange. Because the vocabulary is small and reads as plain annotation, PromptMN is intended to make structured intent more approachable beyond specialist users, including analysts, managers, developers, and everyday users of generative tools.

### Practical Implications for the SDLC
PromptMN fits software development especially well because SDLC professionals already think in structured units such as requirements, constraints, plans, interfaces, and outputs. At the same time, the notation lowers the entry cost for less technical stakeholders because it works like a lightweight template for what a good prompt should contain. In the scenarios examined here, the same vocabulary described a fresh codebase, an update to an existing system, an architectural redesign, an SRS-style prompt, and a mixed natural-language request with only a few directives attached. A full PromptMN block can guide a complex task, while single directives such as %aware, %mustnot, or %showplan can strengthen an ordinary request when a full block would be unnecessary.
The demonstrated results indicate that PromptMN is already recognizable to current advanced models. Empirical testing indicates that contemporary models, such as Claude Fable 5, Claude Opus 4.8, Gemini 3.1 Pro, and GPT-5.5, reliably parse and execute PromptMN directives out of the box, and this confirms that the syntax is viable without the overhead of specialized fine-tuning. The Snake game example further shows how a surfaced plan can be reviewed before code or state changes are produced. For AI-assisted development, this matters because early visibility can reduce needless iteration and prevent an incorrect interpretation from becoming rework later in the chain.
PromptMN also pairs naturally with reverse prompt engineering. When a model restates a desired outcome as a PromptMN prompt, the result becomes a structured artifact that separates role, goal, requirements, constraints, format, and output expectations. Comparing a user’s original prompt with the model’s PromptMN-style reverse prompt can reveal missing assumptions, weak framing, or constraints the model failed to infer. This makes reverse prompting more useful when the prompt will be reused by another person, another model, or an agentic workflow.

### Limitations and Future Work
The results in this paper should be read as demonstrative rather than conclusive. PromptMN has been shown to work in selected examples and SDLC-oriented scenarios, but it has not yet been validated through a controlled benchmark, a user study, or a large-scale comparison against plain prompting, SRS-style prompting, and programming-style pseudocode prompting. Its current interpreter is also probabilistic. Unless a model is fine-tuned or a runtime enforces the specification, correct interpretation depends on the model, the context window, and the prompt environment.
Prompt injection is also worth noting: if PromptMN directives appear in a system prompt and user input is concatenated into the same context, a user could insert keywords to alter the intended behavior. Input sanitization and clear separation between system and user context are practical mitigations that future work should evaluate.
There is also a practical cost. If the full PromptMN specification in Appendix A must be attached to every prompt, it consumes additional tokens. Possible responses include fine-tuning a model on the specification, using a concise model-facing version in Appendix B, or building a linker that attaches only the definitions of the keywords used in a particular prompt. These approaches could reduce token overhead, but they depend on model capability and should be evaluated carefully rather than assumed to be equivalent to the full specification.

## Key insights
- Keywords are inline directives prefixed with %, resolved semantically (not by source order). Authors may write keywords in any order/position; interpreter reorders by meaning before executing.
- Default forms: %keyword \<text>; or %keyword {statement; statement;...}. Special forms noted per keyword.
- Program boundary: $\infty$... $\infty$ encloses the full token set; tokens comma-separated; each token has a distinct meaning the interpreter must apply.
- %keyword: generic notation for a unique command adding instructional info. May appear multiple times, any position; interpreter resolves by semantics, not order.
- {... } block scope: groups statements belonging to the immediately preceding command. Tokens { and } are the open/close. Form: %keyword {stmt; stmt;...}.
- \\n newline: line break; a new term/expression begins after it.
- \<condition>: something evaluable as true/false or satisfied/not (simple declaration or logical expression).
- Angle-bracket tokens (\<condition>, \<name>, \<expression>) are meta-syntactic placeholders, NOT keywords.
- %role: assigns persona(s); all later directives interpreted through this lens. %role \<role>; or %role {r1; r2;}. Ex: %role senior software architect;
- %intro: background/contextual overview of problem, domain, or scope.
