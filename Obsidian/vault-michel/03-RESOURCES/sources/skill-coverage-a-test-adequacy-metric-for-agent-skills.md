## Tese central
Agent skills encode reusable procedural knowledge that guides large language model (LLM) agents across tasks and execution contexts. Existing evaluations primarily assess skills through task-level outcomes, yet task success alone does not reveal which parts of a skill have been exercised or which remain untested. We introduce *skill coverage*, a test adequacy metric that treats the skill artifact as the object under test.

## Argumentos principais
### 1 Introduction
Agent skills package procedural knowledge in a structured form that can augment LLM agents at inference time [^1] [^11]. Studies of public skill ecosystems and cumulative skill creation further show that skills are emerging as reusable artifacts that can be shared across agents and tasks [^7] [^22]. Benchmark evidence also shows that such skills can improve downstream task utility across diverse domains [^11] [^25] [^14].
Although recent benchmarks [^11] demonstrate that skills can improve agent performance, they primarily provide a task-level signal of whether a skill helps an agent complete a particular task. However, task success does not indicate which instructions, expectations, or action patterns encoded in the skill were actually exercised during execution. This distinction matters because a skill is intended to be reused across tasks, models, and execution contexts. To assess whether a skill has been adequately tested, users need evidence about which parts of the skill have been exercised and which expected behaviors remain unobserved under a specific model-task setting. Such evidence enables users to make informed decisions about whether to reuse a skill for a new task, how to adapt it to a new context, and where the skill should be improved.
To provide a more informative signal for skill evaluation, we introduce a testing framework centered on skill behavior constraints. The framework first extracts and formalizes source-grounded behavioral constraints from natural-language skill documents, making explicit the scenarios and expected agent behaviors encoded in the skill. It then defines when such a constraint is covered by an agent trajectory, requiring observable evidence that the trajectory instantiated the relevant scenario and made the constrained behavior judgeable. These two components support skill coverage, a new adequacy perspective that reports which documented behaviors of a skill have been exercised by a test suite under a binary cover/not-cover judgment.

### 2.1 From Skills to Skill Behavior Constraints
A reusable skill is not merely a one-off prompt fragment for completing a single task. Rather, it is a natural-language artifact that guides how an agent should behave across a class of future tasks. Such guidance may describe what the agent should do, what it should avoid, which workflow steps it should follow, how it should use a documented tool path, or what properties its delivered artifacts should satisfy. These prescriptions create *skill-scoped behavioral constraints* on the agent.
This view is consistent with the role of requirements in requirements engineering, where requirements capture desired system behaviors and constraints, and are expected to be verifiable and traceable to their sources [^23] [^6] [^8]. However, a skill document is typically not written as a conventional requirements specification. It may mix instructions, examples, tool usage guidance, workflow steps, common mistakes, artifact expectations, and rationale in free-form natural language. Treating the entire skill document as a testing denominator would therefore be too coarse and ambiguous.
We instead define a conservative abstraction called a *Skill Behavior Constraint* (SBC). An SBC is a source-grounded, observable, EARS-normalized condition-action rule derived from a skill document. It specifies what the agent shall do, avoid, or deliver under skill-scoped conditions. The purpose of this abstraction is not to recover the full semantics of the skill document. Rather, it projects the testable portion of the skill into a finite and auditable denominator for coverage analysis.

### 2.2 Skill Behavior Constraint Coverage
A task-level oracle verdict does not by itself characterize test adequacy. Classical coverage asks whether the relevant elements of the artifact under test have been exercised [^5] [^18] [^27], while oracle-aware coverage further requires that the execution expose evidence an oracle can inspect [^16] [^3]. For skill artifacts, this distinction is essential: loading a skill into an agent context does not mean that any particular documented behavior has been tested. We therefore define coverage with two binary checks: whether the constraint applies, and whether the observable trajectory contains enough evidence to judge the constrained behavior. The coverage judgment does not require that the agent satisfy the behavior constraint.

### 3 Method
Our method operationalizes skill coverage as an auditable procedure that maps skill documents and observable agent trajectories to constraint-level coverage judgments. Given a benchmark task, its task-associated skill documents, and the trajectory produced by an agent, the procedure produces two artifacts: an audited set of source-grounded skill behavior constraints and a set of trajectory-grounded coverage labels. The design follows the definitions in Section 2.1 and Section 2.2, but shifts from definitions to procedure: how constraints are extracted from skill text and how each constraint is judged against observable execution evidence.

### 3.1 Overview
The method has two stages. First, the *constraint extraction* stage turns a natural-language skill document into an auditable denominator of SBCs. Second, the *coverage judging* stage compares each SBC against an observable agent trajectory and decides whether the trajectory makes the constraint covered under the binary cover/not-cover rubric. This decomposition separates the two procedural questions needed for skill coverage: what documented behavior the skill requires, and what observable trajectory evidence exposes about that behavior.

### 3.2 Extracting Skill Behavior Constraints
The extraction stage implements the SBC boundary from Section 2.1 through LLM-assisted candidate extraction followed by human audit and repair. We use the LLM as a recall-oriented parser over skill prose, and use human audit to decide which candidates enter the final SBC set.
Given a skill document $S$, we provide the full skill document to the LLM and ask it to extract candidate SBCs directly. The extraction prompt defines the SBC boundary from Section 2.1, requires EARS-style condition-action output, and includes a one-shot example that illustrates the source-block categories and inclusion/exclusion decisions in Table 1. Thus, Table 1 serves as prompt guidance for what kinds of skill text should or should not produce constraints, rather than as a separate preprocessing classifier over segmented blocks.
The LLM proposes one or more candidate SBCs from the full skill document. Each candidate records the supporting source span, applicability condition, constrained behavior, and expected evidence for judging the behavior. The LLM also proposes atomic splits when a passage contains multiple independently judgeable obligations. At this stage, candidates are intentionally over-inclusive: they are hypotheses about testable skill behavior, not the final denominator.

### 3.3 Judging Coverage in Agent Trajectories
The coverage judging stage assigns a coverage label to each constraint–trajectory pair. The input is a constraint $c=\langle C_{c},B_{c},J_{c}\rangle$ and an observable trajectory $\tau$. We first construct $\mathsf{Obs}(\tau)$ by retaining evidence available to an external evaluator: the user request, visible agent messages, tool calls, tool outputs, command invocations, generated code, file diffs, artifact states, logs, and final responses.
The judge follows a two-step decision procedure. First, it checks *applicability*: whether the observable task context and trajectory instantiate $C_{c}$. If the condition is not instantiated, the constraint is labeled $\mathsf{Uncovered}$ for this trajectory. Second, it checks *adjudicable evidence*: whether the trace contains enough observable evidence, according to $J_{c}$, to decide whether the constrained behavior $B_{c}$ was satisfied or violated. If this evidence check is not satisfied, the constraint remains $\mathsf{Uncovered}$ for this trajectory.
When both checks succeed, the judge records the constraint as $\mathsf{Covered}$, regardless of whether the observed behavior satisfies or violates $B_{c}$. Otherwise, the pair remains $\mathsf{Uncovered}$. In contrast, a final response that merely claims compliance is not sufficient unless the final response itself is the object constrained by the SBC. The judgment must be grounded in concrete trace evidence such as a tool call, file state, command output, generated artifact, or cited response content.

### 4 Evaluation
We evaluate skill coverage in two steps. First, we validate whether the LLM coverage judge matches human labels on a sampled subset of constraint–trajectory pairs. Second, we apply the judge to SkillsBench executions and report how much of the SBC denominator is covered by existing benchmark tasks. The evaluation therefore separates judge reliability from the benchmark-scale coverage result.

### 4.1 Experimental Setup
Benchmark. We use SkillsBench [^11] as the evaluation benchmark. SkillsBench is a suitable setting for skill-level adequacy analysis because its skills are intended to provide reusable procedural guidance for classes of tasks, rather than one-off solutions to individual task instances. This makes task success and skill adequacy meaningfully separable: an agent may complete a task while exercising only a subset of the behaviors documented by the associated skills. We report benchmark-scale coverage on 78 source tasks for which we collected task outcomes, observable agent trajectories, and skill-level coverage judgments.
Agent configurations. We evaluate seven agent–model configurations: Gemini CLI with Gemini 3 Flash and Gemini 3 Pro, Codex with GPT-5.2, and Claude Code with Opus 4.6, Opus 4.5, Sonnet 4.5, and Haiku 4.5. Each configuration is evaluated on the same 78 source tasks. We use the benchmark verifier to obtain task-level success outcomes, and then apply our validated coverage judge to the resulting observable trajectories.
Coverage judge. We use DeepSeek V4 Flash as the LLM coverage judge. It receives only the extracted SBC and the observable trajectory evidence, and applies the two-step coverage rubric from Section 3.3. For benchmark-scale reporting we use its binary cover/not-cover label space. Before applying it at scale, we validate the judge against human labels (Section 4.2).

### 4.2 Measurement Validation
The validated judge matches human cover/not-cover labels. Following the validation-before-scale protocol in Section 3.3, we compare the initial DeepSeek V4 Flash judge against human labels on the 350-case validation sample. The judge agrees with the human cover/not-cover audit on 328 cases, for an exact agreement of 93.71% and Cohen’s $\kappa=0.874$. We therefore use the binary cover/not-cover label space for benchmark-scale coverage analysis.

### 4.4 Analysis and Discussion
Implication for benchmark interpretation. The results suggest that task-level scores should be interpreted as related to, but not sufficient for, skill-level test adequacy. Figure 3 shows that successful executions tend to cover more SBCs than unsuccessful executions, so the benchmark verifier is not unrelated to skill-behavior exercise. However, the same figure also shows that task success leaves a large part of the denominator untested, and that unsuccessful executions can still exercise substantial portions of the skill. A successful task establishes that the agent satisfied the benchmark verifier for that task instance; it does not establish that the execution exercised the workflows, tool-use conditions, prohibitions, or artifact expectations documented by the skill. Skill coverage makes this gap explicit by reporting a denominator over the skill text itself.
Implication for test-suite design. The uncovered portion of the denominator is actionable rather than punitive. Uncovered SBCs identify skill-scoped scenarios or constraints for which the current benchmark executions provide no judgeable evidence. The task-outcome-conditioned view further helps prioritize this work. Successful tasks with low skill coverage reveal benchmark successes that still leave documented skill behavior untested, while unsuccessful tasks with nontrivial coverage may still provide useful evidence about which behaviors were exercised. Benchmark designers can therefore target new tasks, stronger trace capture, or oracle-facing checks at uncovered constraints, while skill authors can identify which parts of a skill have not yet been validated under realistic agent executions.
Scope of the evidence. This evaluation is intended as a preliminary diagnostic application of skill coverage, not as a complete reliability claim about SkillsBench or the tested agents. The judge-validation sample estimates whether the LLM judge matches human labels on sampled constraint–trajectory pairs, while the full SkillsBench results estimate how much of the audited SBC denominator is covered by benchmark executions. The task-outcome-conditioned analysis is associational: it compares coverage across successful and unsuccessful executions, but it does not identify why a particular task outcome occurred. Within that scope, the main conclusion is that task success and skill coverage answer different evaluation questions and should be reported separately.

### 5 Discussion
What skill coverage does and does not claim. Skill coverage is an adequacy signal, not a replacement for task success. A covered constraint means that the trajectory instantiated the relevant condition and exposed enough evidence to judge whether the constrained behavior was satisfied or violated. Conversely, an uncovered constraint should not be interpreted as a violation. It means that the available execution did not create judgeable evidence for that documented skill behavior. The task-outcome-conditioned results reinforce this boundary: successful benchmark executions are associated with higher coverage, but they still leave many SBCs uncovered, and unsuccessful executions can still exercise judgeable skill behavior. The observed coverage gap therefore identifies untested or unjudgeable parts of the skill denominator, not necessarily agent problems.
Using uncovered constraints to improve benchmarks. The practical value of skill coverage is that the uncovered denominator is actionable. If a constraint is uncovered because no task instantiates its condition, benchmark authors can add tasks that exercise that scenario. If the condition is instantiated but the trajectory lacks adjudicable evidence, the benchmark can add stronger trace capture, artifact checks, or oracle-facing assertions. Skill coverage therefore provides an evidence map for improving both skill artifacts and benchmark suites. This evidence map is useful even when task outcomes are available: successful executions with low coverage suggest missing adequacy pressure, while unsuccessful executions with nontrivial coverage can still expose concrete skill behaviors for diagnosis.

### 6 Related Work
Agent skills and skill benchmarks. Agent skills are emerging as reusable natural-language artifacts for extending LLM agents with procedural knowledge, tool-use guidance, and workflow expectations [^1] [^12] [^26]. Recent benchmarks and studies evaluate whether such skills improve downstream task performance, how well agents use skills in realistic settings, and how skills can be generated or accumulated over time [^11] [^25] [^14] [^7]. Other work audits open skill ecosystems and highlights the growing importance of skill quality as skills become shared across agents and tasks [^22]. These efforts establish that skills are practically useful and worth benchmarking. Our work addresses a complementary question: once a reusable skill is evaluated through benchmark executions, which documented behaviors in the skill have actually been exercised with judgeable evidence? Our task-outcome-conditioned analysis further shows why this question is not answered by task outcome alone: task success is related to coverage, but it does not exhaust the documented skill denominator.
Requirements, traceability, and test adequacy. Our formulation builds on the view that requirements should be verifiable, traceable, and connected to desired system behavior [^23] [^6] [^8]. Because skill documents are informal and heterogeneous, we use EARS-style controlled natural language as a lightweight normalization layer for extracting source-grounded behavioral constraints from skill prose [^15]. The coverage side of our work follows the testing tradition that adequacy criteria should describe what a test suite has exercised, rather than only reporting individual test outcomes [^5] [^18] [^19] [^27]. Requirements-based coverage further motivates reporting a denominator over specified behavior rather than over incidental execution artifacts [^20]. We adapt these ideas to agent skills by treating the skill document as the artifact under test and agent trajectories as the evidence surface. This framing separates verifier success from source-grounded skill behavior coverage: one asks whether a task outcome was achieved, while the other asks which documented behaviors received judgeable trajectory evidence.
Oracle-aware coverage and LLM judges. Skill coverage also relates to the oracle problem: an execution is useful for testing only when it exposes evidence that can be checked by an oracle [^3]. Checked coverage formalizes a similar idea for programs by asking whether executed elements influence oracle-checked values [^16]. We reinterpret this principle for natural language skill constraints: a documented behavior is covered only when the trajectory instantiates the relevant condition and contains observable evidence sufficient to judge the constrained behavior. At benchmark scale, this requires many fine-grained attribution decisions. Prior LLM-as-a-judge work shows that automatic judges can be useful when first validated against human or expert judgments [^24] [^13] [^4]. Fine-grained rubric-based evaluators further show how LLM judges can apply predefined criteria at scale [^21] [^9] [^17] [^2]. Our evaluation follows this validation-before-scale pattern by using LLM-based coverage attribution only after checking binary cover/not-cover agreement against human labels.

### 7 Conclusion
This paper addresses the gap between task-level agent evaluation and artifact-level skill adequacy. We introduced skill behavior constraints as source-grounded, observable behavioral units extracted from natural-language skill documents, and defined skill coverage as the fraction of those constraints made judgeable with oracle-adjudicable trajectory evidence. The key idea is to treat the reusable skill document as the artifact under test, rather than assuming that a successful benchmark task has exercised the skill broadly.
Our SkillsBench study shows that this distinction is empirically important. After validating binary cover/not-cover attribution against human labels, we applied the fixed judging protocol to 78 source tasks across seven agent–model configurations and three traces per configuration. Existing executions cover only 39.90–43.98% of the extracted skill behavior denominator on average. Successful task executions cover more of the denominator than unsuccessful executions, but even successful executions leave large portions of documented skill behavior untested. These results show that task success and skill coverage answer different evaluation questions and should be reported separately.
Skill coverage turns untested parts of a skill into inspectable benchmark evidence. It can help benchmark designers identify missing scenarios, help skill authors locate underspecified or unvalidated behaviors, and help users interpret whether a reusable skill has been exercised under realistic agent executions. Future work should extend the protocol to additional skill ecosystems and connect coverage-guided task generation to automatic benchmark improvement.


## Key insights
- We formulate skill-level test adequacy as an artifact-level evaluation problem for reusable natural-language agent skills, exposing the missing behavioral denominator in task-level benchmark results.
- We introduce skill behavior constraints as source-grounded, observable behavioral units extracted and formalized from skill documents while filtering out background prose and incidental implementation detail.
- We define skill coverage over observable agent trajectories, using a binary cover/not-cover judgment to report whether a skill behavior has been exercised with judgeable evidence.
- We validate LLM-based coverage attribution against human labels and apply the resulting protocol to SkillsBench, showing that existing executions cover less than half of the extracted skill behavior denominator.
