## Tese central
Recent advances in large language models (LLMs) have enabled vibe coding, an emerging software development paradigm in which users create applications primarily through natural-language interactions with AI agents. Due to its low barrier to entry, vibe coding is rapidly gaining adoption in practice. Unlike conventional AI-assisted programming, where developers remain responsible for implementation and code review, vibe coding delegates a substantial portion of development to AI systems.

## Argumentos principais
### 1 Introduction
Recent advances in large language models (LLMs) are fundamentally reshaping how software is developed [^11]. What began as code-completion assistants [^22] [^25] capable of generating individual functions or features has rapidly evolved into increasingly autonomous AI coding agents [^19] [^4] that design, implement, and deploy complete, runnable software applications from natural language instructions. This evolution has given rise to a new development paradigm commonly referred to as vibe coding [^13] [^47]. In this paradigm, users describe high-level requirements in natural language and rely on AI coding agents or vibe-coding platforms to generate entire software applications from scratch, with minimal human intervention. Since its introduction, vibe coding has rapidly gained traction across the software development ecosystem, evolving from an experimental concept into a production reality. The appeal of vibe coding lies in substantially lowering the technical barrier to application creation by abstracting away much of the implementation process. As a result, individuals with limited programming experience can now create and deploy applications that previously required significant development expertise. Platforms such as Claude Code [^4] and Lovable [^19] have attracted millions of users and enabled the creation of applications at an unprecedented scale. For example, Lovable reported more than eight million users and over 100,000 new applications created every day as of early 2026 [^31]. Moreover, more than 10% of the applications are deployed as live, publicly accessible systems, as we later show in our study. With its popularity, vibe coding raises a critical concern: *how (in)secure are vibe-coded applications in practice?* Unlike traditional software development, vibe coding also allows non-expert individuals to create and directly deploy applications without fully understanding the underlying implementation. Therefore, security-critical decisions made by AI coding agents may remain unaudited. While modern agents are increasingly designed to incorporate secure programming practices, such as authentication mechanisms, input validation, and common security safeguards [^18] [^3], practitioners continue to report applications that expose sensitive credentials, implement flawed access-control logic, construct vulnerable database queries, or deploy insecure configurations [^8] [^29] [^43]. More importantly, as vibe coding scales, a single vulnerability generated by coding agents may be replicated across thousands of applications. Although a growing body of work has examined the security of LLM-generated code [^32] [^17] [^47] [^6], existing studies fall short of capturing the security risks of vibe-coded applications. Prior research has primarily focused on evaluating the security of AI-assisted programming. In this paradigm, AI coding agents operate within established codebases and are tasked with completing specific development activities, such as fixing bugs or implementing new features [^46] [^45], where software development is largely reduced to mere code-filling tasks. This paradigm fundamentally differs from vibe coding, where AI coding agents are instructed to create entire software applications from scratch. In vibe-coded applications, security vulnerabilities may arise not only from insecure code generation, but also from flawed architectural choices, poorly defined trust boundaries, and insecure integrations among components selected and assembled by LLM agents. Therefore, findings from prior studies on LLM-generated code cannot be generalized to characterize the security of vibe-coded applications.
#### Our Work
To this end, we conduct the first systematic study of the security of vibe-coded applications. We construct VibeApps, a large-scale corpus of 10,517 open-source vibe-coded applications. From this corpus, we randomly sample 200 deployed applications that are publicly accessible and use them as the target of our security analysis. We then develop a human-in-the-loop vulnerability-discovery workflow, combining agent-assisted candidate discovery with exploitability triage and manual validation. Using this workflow, we identify 1,471 vulnerabilities across the sampled applications, forming our vulnerability dataset VibeVulns. To the best of our knowledge, this is the first large-scale dataset for studying the security of real-world vibe-coded applications. We seek to answer the following research questions:

### 2 Background and Related Work
#### AI-Assisted Programming and Vibe Coding
LLMs are becoming a standard tool in modern software development. Current coding assistants support a wide range of tasks, including code completion, bug fixing, and even repository-level modification. We refer to this development paradigm as *AI-assisted programming* [^11]: developers remain responsible for application design, implementation decisions, and code review, while LLMs provide code-level assistance. Industry-grade tools such as GitHub Copilot are now used by millions of developers [^38], leading to a substantial fraction of code in production repositories being generated with LLM assistance [^5]. Building on these advances, *vibe coding* has emerged as an AI-led development paradigm. Here, users describe desired requirements for applications in natural language, and LLM agents autonomously design, implement, and iteratively refine code to produce complete, runnable applications with little or no human inspection [^7]. Unlike AI-assisted programming, where humans remain the primary authors and decision-makers, vibe coding shifts substantial authorship and oversight toward AI agents, leaving humans primarily specifying high-level goals and providing feedback. This distinction is central to our study because security-relevant decisions in vibe-coded applications may be made as part of an AI-led development workflow rather than through explicit human design and review.

### 3 Dataset Construction
In this section, we construct a high-quality vulnerability dataset of vibe-coded applications to investigate their security risks. To this end, we first construct VibeApps, a representative corpus of real-world vibe-coded applications. We then develop a vulnerability-discovery framework that combines agentic code auditing with human-in-the-loop validation to discover their vulnerabilities. The resulting vulnerabilities constitute the dataset VibeVulns, which serves as the foundation for our security analysis.

### 3.1 Vibe-Coded Application Collection
To collect real-world vibe-coded applications, the primary challenge lies in how to identify such applications, as they are not explicitly labeled as vibe-coded. To address this, we design a three-stage pipeline. We first discover potential vibe-coded applications using fingerprints associated with AI coding agents. We then filter these candidates according to a set of criteria to obtain a high-quality subset. Finally, we determine applications to be vibe-coded based on the degree of AI contributions.
#### Stage 1: Potential Application Discovery
Although there are no explicit indicators for vibe-coded applications, the repositories that have AI-generated code usually contain some fingerprints left by the corresponding coding agents. For example, repositories using Claude Code include the configuration directory.claude/, while those developed with Lovable contain metadata \<meta name="author" content="Lovable" />. These fingerprints provide useful signals for discovering potential vibe-coded applications. Based on this observation, we query the GitHub Code Search API to collect the repositories from GitHub that contain the relevant fingerprints. In this study, we focus on two popular AI coding agents: Claude Code, representing command-line-based agents, and Lovable, representing AI-based development platforms. They capture distinct development workflows and provide broader coverage of the current vibe-coding ecosystems. Although we leverage the fingerprints of these two coding agents to discover candidate applications, in practice, many repositories are developed using multiple AI coding agents. Therefore, our collected applications are not limited to those developed exclusively by Claude Code or Lovable. This step results in an initial dataset of 74,800 candidate applications.

### 3.2 Vulnerability Discovery
After collecting vibe-coded applications, we construct their vulnerability dataset for security analysis. However, discovering vulnerabilities across these applications poses another challenge: they vary substantially in *scale*, *programming language*, *technology stack*, and even *application type* (as shown in Section˜4). No conventional vulnerability-detection tool can effectively cover such a diverse set of applications. To address the challenges introduced by different *scales*, *programming languages*, and *technology stacks*, we adopt code-auditing agents for vulnerability discovery. Nevertheless, auditing different *types* of applications requires different agent capabilities and auditing workflows. Therefore, we focus on the dominant type of vibe-coded applications (from Section˜4): web applications. Among the 10,517 collected applications, 9,935 (over 94%) are web applications, making their vulnerabilities broadly representative of the overall security landscape of vibe coding. Based on this, we develop the vulnerability-discovery framework that combines multi-agent code auditing with structured human-in-the-loop validation. Specifically, we first leverage a multi-agent system to analyze each repository and generate structured vulnerability reports. We then perform both AI-assisted and manual validation to eliminate redundancy and false positives, producing the final set of vulnerabilities for analysis.
#### 3.2.1 Multi-Agent Code Auditing
We leverage LLM-based code auditors to identify potential vulnerabilities in each repository. To maximize vulnerability coverage, we adopt a $2\times 2$ auditing design that combines two agent frameworks with two security auditing pipelines, yielding four independent audit reports for each repository. The two agent frameworks are Claude Code with Claude Sonnet 4.6 and GitHub Copilot with GPT-5.3-Codex. Each agent is paired with two complementary security skill sets: OpenAI’s official security-best-practices skill [^26], which provides guidance for detecting web application vulnerabilities across a range of technology stacks, and Antigravity Awesome Skills [^39], a widely adopted collection of security skills with nearly 40k GitHub stars that integrates SAST tools and vulnerability-detection rules. Each skill set is organized around a primary security-analysis agent and a collection of technology-specific subagents. The primary agent, defined by the top-level SKILL.md, first analyzes the repository structure, framework usage, and technology stack. Based on this assessment, it automatically invokes specialized subagents tailored to the detected technologies. These subagents implement security auditing strategies specific to individual frameworks and ecosystems, such as React and Next.js. This hierarchical architecture helps reduce missed vulnerabilities arising from the limitations of any single model or auditing strategy. We apply this $2\times 2$ design to audit the web applications. To ensure that the discovered vulnerabilities reflect realistic security exposure, we restrict our analysis to applications with publicly accessible deployments. These account for 1,170 of the 9,935 web applications (nearly 12%, as shown in Section˜4). Auditing all deployed applications would require substantial computational resources and manual validation effort. Therefore, we randomly sampled 200 applications from the 1,170 deployed repositories for detailed security auditing. As a result, across these applications, the four auditing configurations generated 9,353 raw vulnerability reports. The auditing process consumed 7.98 billion tokens and 130 hours of compute time, averaging 9.98 million tokens and 9.7 minutes per audit run.

### 4 RQ.1: What Is the Landscape of Real-World Vibe-Coded Applications?
To answer RQ.1, we characterize the 10,517 repositories in our corpus VibeApps along four dimensions: repository scale and development characteristics, application categories, technology stacks, and deployment status. This characterization establishes the landscape of real-world vibe-coded applications and motivate our subsequent security analysis of deployed web applications.
#### Repository Scale and Development Characteristics
Real-world vibe-coded applications are often large-scale software projects. The median repository contains 8,351 lines of code (LoC) across 101 files, while repositories at the 90th percentile reach 22,714 LoC and 212 files. Overall, 94.5% of repositories contain more than 5,000 LoC, and 36.0% exceed 10,000 LoC. Despite their considerable scale, these applications are typically developed within a short time. Measured from the first to the last commit, the median development span is 9.8 days. Furthermore, 66.3% of repositories have development spans shorter than 30 days, and 84.3% are completed within 90 days. However, these short development spans do not imply limited development activities: 65.1% of repositories contain at least 25 commits, indicating substantial development activity within compressed periods. To better characterize the development process, we examine the relationships between repository scale, development span, and commit counts. Repository scale is moderately correlated with commit count (Spearman $\rho=0.67$ for LoC and $\rho=0.62$ for file count), suggesting that larger repositories are generally associated with more intensive development activity. In contrast, commit count is more weakly correlated with development span (Spearman $\rho=0.51$), indicating that elapsed time explains comparatively little of the variation in development effort. Moreover, commit density declines sharply as development span increases (Spearman $\rho=-0.65$). The shortest-span quartile of repositories sustains a median of 11.9 commits per day, whereas the longest-span quartile drops to 1.2 commits per day, indicating that vibe-coded applications are typically built in short, high-intensity periods. Taken together, these findings reveal a distinctive scale-and-speed profile: many vibe-coded applications combine large codebases with highly concentrated development activity. Rather than emerging through the gradual accumulation of small code generations, they are often produced through intense bursts of project-level iteration. This development pattern indicates that few human reviews and hardening are likely to be applied before deployment. This confirms our motivation to study the security of vibe-coded applications.

### 6 RQ.3: What Factors Contribute to These Vulnerabilities?
To understand why vulnerabilities arise in vibe-coded applications, we qualitatively analyze each vulnerability. For each case, two authors independently inspect the vulnerable code, the commit history that produced it, and any surrounding inline comments. Each reviewer then assigns an open-ended label describing why the vulnerability was introduced, rather than forcing cases into a predefined taxonomy. The two authors subsequently consolidate their labels through discussion, merging overlapping tags and refining boundary cases. Through this, we identify eight recurring *failure modes*, and group into three broader *defect classes*: memory defect (Section˜6.1), objective defect (Section˜6.2), and knowledge defect (Section˜6.3), as illustrated in Figure˜3. Furthermore, we map the failure modes into the vibe-coding lifecycle, which comprises four development stages. In the *specification* stage, the user provides instructions, and the agent plans the solution architecture. In the *implementation* stage, the agent generates code for the intended design. In the *iteration* stage, the user and agent repeatedly test, debug, and revise the codebase. In the *deployment* stage, the application is released to production and exposed to real users. These stages are not strictly linear and may recur as the user and agent repeatedly refine the application. We also report the number and severity of vulnerabilities in each failure mode. We show how failure modes lead to the vulnerabilities across the OWASP Top 10 categories in Figure˜4. We next elaborate on each failure mode.

### 6.1 Memory Defects
Memory defects arise when security obligations introduced earlier in development are not reliably retained, propagated, or revisited as the project evolves. Although coding agents can reconstruct context from prompts, files, and recent interactions, they do not maintain a complete and reliable memory of all project-wide security requirements over time. In the vibe-coding lifecycle, these defects are primarily in later-stage development, especially during *iteration*, when security-relevant changes must be propagated consistently across the codebase, and *deployment*, when deferred obligations must still be completed before release. Together, these defects account for 287 (19.5%) vulnerabilities.
#### Iteration Stage: \\scriptsize1⃝ incomplete change propagation
This failure mode occurs when the agent applies a security requirement in one part of the codebase but fails to propagate to other relevant locations. Sometimes, this manifests as a “forward propagation failure”, where a newly introduced security property is applied only to new code and never back-fitted to pre-existing paths. In other cases, it manifests as a “backward conformance failure”, where a new code path is added that skips a security property the rest of the codebase already enforces. The following code snippet in Figure˜5 illustrates the forward variant in FrameOps [^20]. A validateApiKey middleware is added for a newly introduced router, but the same protection is not applied to 11 pre-existing handlers. As a result, those handlers remain reachable without the intended authorization check. We identify 185 vulnerabilities (12.6%) caused by this failure mode. Their severity is also substantial. 77.3% are rated Critical or High. These vulnerabilities are especially common in Broken Access Control and Injection, where the same protection often needs to be applied consistently across multiple endpoints, handlers, or sinks.

### 6.2 Objective Defects
Objective defects occur when immediate functional goals are prioritized over long-term correctness or security during development. In these cases, security is systematically subordinated to short-term objectives such as making the application demo-ready, unblocking a visible error, or restoring functionality as quickly as possible. Objective defects are most prevalent in the *specification* and *iteration* stages. In total, they account for 450 (30.6%) vulnerabilities.
#### Specification Stage: \\scriptsize3⃝ demo-oriented design
This failure mode occurs when the application is designed around immediate usability, convenience, or demonstrability rather than secure-by-default practice. In these cases, insecure mechanisms are not merely left to be fixed later. Instead, they are incorporated into the intended design. Examples include dummy encryption functions, static demo keys, or placeholder authorization logic that remains acceptable as long as the application appears to work. Figure˜7 illustrates this pattern in fuyou [^12]. The application stores secret OAuth tokens in front-end localStorage after applying only base64 encoding, with the choice described as suitable for demonstration. This exposes sensitive credentials to client-side access. Demo-oriented design accounts for 237 vulnerabilities (16.1%), mainly in Broken Access Control and Cryptographic Failures. These vulnerabilities often lead to authentication bypasses or secret exposure, and 84.0% are classified as Critical or High.

### 6.3 Knowledge Defects
Knowledge defects refers that both agent and user may fail to recognize the security requirements of the application during its development. In the vibe-coding workflow, many security constraints are not explicitly stated, and agents may not derive them from the surrounding application context. On the user side, developers may overlook necessary security steps. These knowledge gaps on both sides result in insufficient, incorrect, or fabricated security practices. This is the largest defect class, accounting for 734 vulnerabilities (49.9%). Knowledge defects span the *specification*, *implementation*, and *deployment* stages.
#### Implementation Stage: \\scriptsize5⃝ hidden security rules
This captures cases where generated code violates a security requirement that is necessary for safe implementation but is not explicit in the user request. Such requirements include hashing credentials before storage, validating input before sensitive sinks, and enforcing security checks on the server side. Since these requirements often do not affect visible functionality, their absence may go unnoticed. The following shows a representative example [^21] in Figure˜9. The vulnerable code serializes an external response and writes it into the page using document.write, but omits the required escaping step, resulting in reflected XSS. Hidden security rules account for 646 vulnerabilities (43.9%), making it the most common reason in our analysis. These vulnerabilities appear most frequently in Broken Access Control and Injection, where a single missing security requirement can affect many routes, handlers, or sinks. Although the violated rules vary in immediate impact, 66.7% of these vulnerabilities are classified as Critical or High.

### 7 RQ4: What Conditions Can Mitigate Vulnerabilities?
To understand when vibe coding is likely to mitigate vulnerabilities, we study three potential conditions that may influence its behaviors: (1) agent harness, (2) underlying language model, and (3) user prompt. To isolate the effect of each condition, we conduct a controlled replay experiment that re-executes the original vibe-coding scenarios in which vulnerabilities were introduced and measures whether the agent reintroduces them under a different configuration.
#### Experimental Setup
We select three representative repositories from our vulnerability dataset: my-board-app [^15] for knowledge defects, qart-nfc-production [^23] for objective defects, and PhantomOS [^34] for memory defects. From each repository, we select five vulnerabilities that have the feasibility of reproduction and collectively cover all failure modes within the corresponding defect category, which results in a total of 15 targets to replay. These targets cover seven failure modes overall. We only exclude hallucination as it is difficult to reproduce reliably. Details are shown in Appendix C. For each target, we reconstruct the pre-state in which the vulnerability has not been introduced yet, while the surrounding feature context remains intact. To achieve this, we either roll back to the parent commit or surgically remove only the code change that introduces the vulnerability when the original commit contained significant feature changes that affected the whole context. After that, we manually issue a task prompt that requests the same functionality as the original vulnerability-introducing scenario. We then run the agent to complete the task and collect both the generated artifacts and traces of the agentic process. Using these artifacts, we manually evaluate whether the target vulnerability has been reintroduced. We flag it as triggered if the same or similar vulnerability reappears; otherwise not.

### 8.1 Sources of Insecurity
Taken together, our findings suggest that insecurity in vibe coding emerges from the interaction of three conditions: agent-level limitations, workflow-level weaknesses, and responsibility misalignment.
#### Agent-Level Limitations
Many of the defects above can be traced to the behavior of the agent. First, because LLMs are effectively stateless across turns, they rely on reconstructed context, which is often incomplete and can introduce inconsistency (memory defects). Second, agents are optimized to follow user instructions and produce visible progress, which can deprioritize long-term security concerns (objective defects). Third, model knowledge is incomplete, and generation is probabilistic, which can produce incorrect claims and omitted implicit security rules (knowledge defects). These limitations are structural and cannot be fully resolved by model scaling or prompt engineering alone.

### 8.2 Implications for Mitigation
Securing vibe-coded applications requires a shift from model-centric improvement to workflow-aware mitigation. Stronger models can reduce some knowledge defects, but they do not reliably address memory defects or objective defects. Therefore, mitigation requires a systematic approach that addresses the entire system.
#### Security as a Workflow Property
Security should be treated as a property of the workflow, not only of the generated code. Simply incorporating security scanning into the code generation step is insufficient. Future vibe-coding systems should integrate security controls at each stage of the lifecycle, including requirements specification, code generation, and deployment configuration.

### 9 Conclusion
In this paper, we study the prevalence, severity, and categories of vulnerabilities in vibe-coded applications, and investigate whether those vulnerabilities are caused by systematic failures in the vibe-coding workflow. Our study of 1,471 vulnerabilities from real-world, publicly deployed vibe-coded applications shows that (1) vulnerabilities are widespread, with 90.0% of repositories containing at least one vulnerability; (2) these vulnerabilities are often severe, with over three-quarters of the vulnerabilities rated as high or critical; and (3) they are concentrated in security-critical parts of the application stack. We identify eight recurring failure modes that introduce vulnerabilities throughout the vibe-coding development lifecycle and explain how security obligations are forgotten, underspecified, or never established. Our study suggests that securing vibe-coded applications requires treating vibe coding as a workflow-level security problem more than a code-generation problem.


## Key insights
- RQ.1: What is the landscape of real-world vibe-coded applications? We characterize the ecosystem of vibe-coded applications, including their scales, development characteristics, and application categories, providing a foundation for subsequent security analysis.
- RQ.2: How prevalent and severe are vulnerabilities in deployed vibe-coded applications? We measure the prevalence, severity, and distribution of their vulnerabilities to assess their overall security posture.
- RQ.3: What factors contribute to these vulnerabilities? We investigate the root causes of identified vulnerabilities and uncover recurring factors that lead to vulnerabilities.
- RQ.4: What conditions can mitigate the vulnerabilities of vibe-coded applications? We examine whether advanced agent harnesses, stronger foundation models, and carefully-crafted user prompts can mitigate vulnerabilities in generated applications.
- We construct the first large-scale dataset of 10,517 real-world vibe-coded applications. The dataset captures a diverse ecosystem of AI-generated software and serves as a foundation for future research in this domain.
- Leveraging this dataset, we present the first large-scale empirical study of security vulnerabilities in vibe-coded applications, providing a systematic characterization of the prevalence, distribution, and patterns of security risks in vibe-coding software development.

## Minha Síntese
**O que muda:** Vibe-coded apps têm 90% de vulnerabilidades, com 76% rated high/critical — a insegurança emerge da interação de memory defects, objective defects e knowledge defects no workflow, não apenas do código gerado.

**Conexão pessoal:** Ao usar agentes para gerar código no vault ou projetos, preciso tratar segurança como propriedade do workflow, não do código — cada estágio (spec, implementação, deploy) precisa de controls.

**Próximo passo:** Adicionar um security checklist no pipeline de ingestão para código gerado por agentes, focando em hidden security rules e incomplete change propagation.
