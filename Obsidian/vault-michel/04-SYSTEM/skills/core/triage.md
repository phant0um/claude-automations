# Skill: Triage

## Propósito

Sistema state-machine para mover issues e PRs externos por categorização, verificação, briefing e routing. PRs são tratados como "issue com attached code" — mesmos roles, mesmos states.

---

## Condições de Ativação

Ative quando:
- User pede "triage" ou "show me what needs attention"
- Issue/PR novo precisa categorização
- Bulk triage de backlog

NÃO ative para: issues já triaged; features em implementação ativa.

---

## Roles

### Category Roles (exatamente um obrigatório)
- `bug` — algo está quebrado
- `enhancement` — nova feature ou melhoria

### State Roles (exatamente um obrigatório)
- `needs-triage` — maintainer precisa avaliar
- `needs-info` — esperando reporter por mais informação
- `needs-info` → retorna para `needs-triage` quando reporter responde
- `ready-for-agent` — fully specified, ready para AFK agent
- `ready-for-human` — precisa implementação humana
- `wontfix` — não será actioned

**Para PRs:** `ready-for-agent` = brief attached, agent takes next step; `ready-for-human` = ready para human merge.

> *This was generated by AI during triage.* — disclaimer obrigatório em todo triage comment.

---

## Protocolo

### 1. Show what needs attention
Three buckets (oldest first):
1. **Unlabeled** — nunca triaged
2. **needs-triage** — evaluation in progress
3. **needs-info com reporter activity** — needs re-evaluation

### 2. Gather Context (per issue/PR)
- Ler full body, comments, labels, author, dates (diff para PRs)
- Parse prior triage notes (não re-ask resolved questions)
- Explorar codebase usando domain glossary, respeitando ADRs
- **Redundancy check**: buscar implementação existente por domain concept → se found, wontfix
- **Prior rejection check**: ler .out-of-scope/ para requests similares rejeitados

### 3. Recommend
- Category + state recommendation com reasoning
- Brief codebase summary relevante
- **Wait for maintainer direction**

### 4. Verify the claim
- **Bug**: reproduzir dos steps do reporter
- **PR**: checkout diff, rodar testes relevantes
- Report: confirmed (com code path), failed, ou insufficient detail (strong needs-info signal)
- Verificação confirmada = agent brief muito mais forte

### 5. Grill (if needed)
- Rodar `/grill-me` + `/decisions` juntos
- 1 pergunta por vez, sharpening domain terms
- Update decisions.md/ADRs inline conforme decisões landam

### 6. Apply the outcome

| State | Action |
|-------|--------|
| `ready-for-agent` | Post agent brief comment |
| `ready-for-human` | Mesma estrutura, note why can't delegate |
| `needs-info` | Post triage notes (established + questions) |
| `needs-triage` | Apply role, optional comment |

**wontfix sub-cases:**
- Already implemented → point to where it lives, NÃO escrever .out-of-scope/
- Rejected (bug) → polite explanation, close
- Rejected (enhancement) → escrever .out-of-scope/, link from comment, close

### Quick State Override
Se maintainer diz "move #42 to ready-for-agent" → trust, apply directly. Skip grilling. Perguntar se quer agent brief.

## Completion

- [ ] Todos items needs-attention triaged (unlabeled, needs-triage, needs-info com activity)
- [ ] Cada item tem: category + state + reasoning
- [ ] Redundancy check executado (implementação existente?)
- [ ] Prior rejection check executado (.out-of-scope/)
- [ ] Bug reproduzido ou PR diff verificado
- [ ] Triage notes posted para needs-info
- [ | AI disclaimer em todo triage comment

## Failure modes

- **Re-ask resolved**: perguntar algo já respondido em prior triage notes → sempre ler notes primeiro
- **Wontfix without .out-of-scope**: rejeitar enhancement sem salvar em .out-of-scope/ → salvar para futura referência
- **Skip redundancy**: não checar se já implementado → wontfix com pointer é mais barato que triage completo
- **Skip verification**: não reproduzir bug ou não testar PR → verificação confirmada fortalece brief
